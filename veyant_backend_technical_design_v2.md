**Target audience:** Travel tech CTOs evaluating technical PM capabilities

## 1. What This Is (and What It Isn't)

This is a portfolio piece that demonstrates how a technical product manager with deep travel industry domain expertise approaches building an AI orchestration platform. It is NOT a production backend. It IS:

- A working frontend demo showing the Discovery Bridge concept (already built)
- A clean architecture design that could be handed to an engineering team
- A proof-of-concept policy engine that validates the domain model with real tests
- A professional GitHub repo with CI/CD and automated deployment to Azure
- Optionally, an MCP server proving the "infrastructure for AI agents" concept

The narrative for interviews: "I own the product architecture and vision. I validate assumptions by prototyping critical domain logic. I produce clean specs that engineering teams can build from. And I use modern tooling to keep the iteration loop tight."

## 3. Build Phases

### Phase 0: GitHub + Azure (session 1, ~1-2 hours)

Goal: Get the existing frontend demo into a professional repo and live on a real URL.

What to do:
1. Create a GitHub repository (veyant-demo or similar)
2. Add the frontend index.html and CLAUDE.md
3. Add a README.md explaining what the project is
4. Set up Azure Static Web Apps connected to the GitHub repo
5. Verify: pushing to main auto-deploys, demo is live at a real URL

What this proves in an interview: "I have a working demo you can see right now at [url]. The repo has CI/CD, so every push deploys automatically."

Tech involved:
- Git basics (init, add, commit, push)
- GitHub (repo creation, personal access token)
- Azure Static Web Apps (free tier, connects directly to GitHub)
- GitHub Actions (auto-generated by Azure setup, or a simple YAML file)

Claude Code can handle all of this. You just need your GitHub credentials and Azure account ready.

### Phase 1: Policy Engine + Domain Types (session 2-3, ~1-2 days)

Goal: Build the one piece of backend logic that proves "this is not a Claude wrapper."

What to build:

**src/domain/types.ts** - Core TypeScript interfaces
- Traveler (profile, loyalty accounts, preferences)
- CorporatePolicy (rules, exceptions, approval thresholds)
- PolicyRule and PolicyCondition (structured rule definitions)
- FlightOption, HotelOption (supplier catalog types)
- PolicyComplianceResult (what the engine returns)

These types are the contract. They document your domain model in executable form. A CTO reading types.ts sees 20 years of travel industry knowledge encoded as data structures: loyalty alliance awareness, market-specific policy exceptions, tiered approval thresholds.

**src/domain/policy-engine.ts** - Deterministic rule evaluation
- Takes: CorporatePolicy + proposed trip changes + total cost
- Returns: PolicyComplianceResult with pass/fail per rule and overall status
- Handles: rate limits by market, exception overrides, approval threshold checks
- Does NOT call Claude or any external service
- Pure function, no side effects, fully testable

**src/domain/policy-engine.test.ts** - Test suite (the money shot)
- Hotel rate under default limit: passes
- Hotel rate over default limit: fails
- Hotel rate over default but London exception applies: passes
- Total trip cost under auto-approval threshold: no approval needed
- Total trip cost over threshold: requires manager approval
- Preferred supplier bonus: flagged as preferred
- Non-compliant cabin class: fails
- Multiple rules, mixed results: overall status correctly derived

**src/data/mock-data.ts** - Seed data as TypeScript constants
- Sarah Chen's full profile (matching the frontend demo exactly)
- Acme Corp's policy rules
- Flight and hotel options for both demo scenarios

What this proves in an interview: "The policy engine is deterministic code with its own test suite. In the full system, when Claude needs to check policy compliance, it calls this as a tool and gets back a typed result. The rules are data, the evaluation is logic. A TMC can audit exactly why a booking was approved. You can't get that from a prompt."

Run `npm test` and 8-10 green checkmarks appear. That's your mic drop.

### Phase 2: MCP Server (session 4-5, ~2-3 days, only if Phase 1 is solid)

Goal: Prove the "Plaid for AI agents" concept with a working MCP server.

What to build:

**src/mcp/server.ts** - MCP server entry point (stdio transport)

**MCP Resources** (read-only context that AI agents can access):
- `traveler://sarah-chen/profile` - returns her full profile from mock data
- `trip://london-sept/details` - returns trip segments
- `policy://acme-corp/rules` - returns corporate policy

**MCP Tools** (actions that AI agents can invoke):
- `check_policy_compliance` - calls the same PolicyEngine from Phase 1
- `search_travel_options` - returns filtered options from mock data
- `get_preference_score` - scores an option against traveler preferences

What this proves in an interview: Open Claude Desktop, connect to the Veyant MCP server, type "Sarah Chen has her London meeting extended to Wednesday. What are her options?" Watch Claude call your tools, check the policy, score the options, and present a recommendation. Live. Then say: "Any AI agent from any TMC or travel platform can connect to this same interface. The intelligence is in the tools, not the model. That's the Discovery Bridge."

### Phase 3: Stretch Goals (only after Phase 2 works)

These are parking lot items. Do not start them until Phases 0-2 are complete and tested:

- Connect frontend to a real API (replace hardcoded steps with API calls)
- Claude orchestration (BRAIN/BRIDGE as separate agent calls with tool use)
- SSE streaming (orchestration steps stream to the frontend in real time)
- Preference model (scoring engine, separate from policy engine)
- Lakebase/Postgres (real database instead of in-memory mock data)
- Supplier connector abstractions (mock NDC/GDS interface)

Each of these can be discussed intelligently in an interview as a planned next phase with a clear design, even if the code doesn't exist yet. The architecture doc covers them.

## 4. Core TypeScript Interfaces (types.ts)

These interfaces are the most important artifact in the repo after the demo itself. They encode domain knowledge.

```typescript
// === MY TRAVEL CUBE ===

export interface Traveler {
  id: string;
  name: string;
  email: string;
  company: string;
  homeAirport: string;
  loyaltyAccounts: LoyaltyAccount[];
  preferences: TravelerPreferences;
}

export interface LoyaltyAccount {
  program: string;          // "SkyTeam", "Marriott Bonvoy", "BA Executive Club"
  tier: string;             // "Platinum Elite", "Club Gold"
  memberId: string;
  pointsBalance: number;
  benefits: string[];       // ["free change", "lounge access", "upgrade eligible"]
  alliance?: string;        // "oneworld", "SkyTeam", "Star Alliance"
}

export interface TravelerPreferences {
  seatPreference: "window" | "aisle" | "no_preference";
  cabinPreference: "economy" | "premium_economy" | "business" | "first";
  preferredCarriers: string[];
  preferredHotelBrands: string[];
  nonStopPreferred: boolean;
}

// === CORPORATE POLICY CUBE ===

export interface CorporatePolicy {
  companyId: string;
  companyName: string;
  rules: PolicyRule[];
  preferredSuppliers: PreferredSupplier[];
  approvalThresholds: ApprovalThreshold[];
  exceptions: PolicyException[];
}

export interface PolicyRule {
  id: string;
  category: "flight" | "hotel" | "general";
  description: string;
  field: string;            // "hotel_nightly_rate", "flight_cabin", "total_additional_cost"
  operator: "lt" | "lte" | "gt" | "gte" | "eq" | "in";
  value: number | string | string[];
}

export interface PreferredSupplier {
  category: "airline" | "hotel";
  supplierName: string;
}

export interface ApprovalThreshold {
  maxAmount: number;
  currency: string;
  autoApprove: boolean;     // true = no manager needed below this amount
}

export interface PolicyException {
  market: string;           // "London", "New York", "Tokyo"
  field: string;            // same as PolicyRule.field
  overrideValue: number;
  reason: string;           // "High cost market exception"
}

// === SUPPLIER CATALOG CUBE ===

export interface FlightOption {
  id: string;
  carrier: string;
  flightNumber: string;
  origin: string;
  destination: string;
  departureTime: string;
  cabin: string;
  price: number;
  changeFee: number;
  upgradeable: boolean;
  upgradeMethod?: "miles" | "status_benefit";
  upgradeCost?: number;
  seatAvailable: string;
}

export interface HotelOption {
  id: string;
  brand: string;
  property: string;
  city: string;
  ratePerNight: number;
  rateType: "corporate" | "rack" | "loyalty";
  available: boolean;
  loyaltyBenefits: string[];
}

// === POLICY ENGINE OUTPUT ===

export interface PolicyComplianceResult {
  overallStatus: "compliant" | "exception_applied" | "requires_approval" | "violation";
  ruleResults: PolicyRuleResult[];
  approvalsNeeded: string[];  // empty if auto-approved
}

export interface PolicyRuleResult {
  ruleId: string;
  description: string;
  status: "pass" | "exception" | "fail";
  detail: string;
}

// === TRIP CHANGE PROPOSAL (input to policy engine) ===

export interface TripChangeProposal {
  segments: ProposedSegment[];
  totalAdditionalCost: number;
  market: string;           // primary market for exception lookup
}

export interface ProposedSegment {
  type: "flight" | "hotel";
  supplier: string;
  rate: number;             // nightly rate for hotel, ticket price for flight
  cabin?: string;           // for flights
  city?: string;            // for hotels, used for market exception lookup
}
```

## 5. Policy Engine Design

The policy engine is intentionally simple. It evaluates structured rules against a proposed trip change. No AI, no external calls, no database. Pure logic.

```
Input:
  CorporatePolicy (Acme Corp rules)
  + TripChangeProposal (what Sarah wants to do)

Processing:
  For each PolicyRule:
    1. Find the relevant field in the proposal (hotel rate, cabin class, etc.)
    2. Compare against the rule's condition
    3. If fails: check if a PolicyException applies for this market
    4. If exception applies: mark as "exception" not "fail"
  Then check ApprovalThresholds:
    If totalAdditionalCost > threshold: requires_approval
    If under threshold: auto-approved

Output:
  PolicyComplianceResult with per-rule breakdown
```

The engine handles a few non-obvious travel industry patterns that show domain depth:

- Market-specific rate exceptions (London hotels cost more than the global default)
- Loyalty-status-based fee waivers (Club Gold = $0 change fee, but that's a supplier benefit, not a policy rule, so the engine doesn't evaluate it, it just passes through)
- Preferred supplier flags (not a hard rule, but surfaced in the result)
- Tiered approval (auto-approve under $2K, manager above, VP above $5K)

## 9. Architecture Docs (for interview discussion, not built)

The following exist as design documents in the repo or in conversation history. They show architectural thinking without implying the code exists:

- **Agentic orchestration design:** BRAIN agent (context tools) and BRIDGE agent (execution tools) as separate Claude calls with distinct system prompts and tool sets. Coordinator manages handoff. BRAIN uses Haiku for retrieval, Sonnet for analysis. BRIDGE uses Haiku for execution, Sonnet for exception handling. Token cost optimization is built into the architecture.

- **Data layer strategy:** SQLite for prototype, Postgres/Lakebase for production. Drizzle ORM for portable data access. Schema maps 1:1 to TypeScript interfaces.

- **Supplier connector abstraction:** Interface for flight/hotel search and booking. Mock implementation for demo. Real implementation would connect to NDC APIs, GDS, and direct supplier portals.

- **MCP as the external interface:** The same tools available to the internal orchestrator are exposed via MCP for external AI agents. This is the "Plaid for AI agents" positioning.

- **Multi-tenant isolation:** Each TMC gets isolated data spaces. Usage metering for API calls and AI inference. Rate limiting per tenant.

- **Security architecture:** Three-tier data classification. Credential vault for supplier logins. OAuth 2.0 for platform auth. Corporate SSO for travelers.

These are all legitimate engineering handoff artifacts. The policy engine + tests prove you can bridge from design to working code. The architecture docs prove you can think at system level.

## 10. What to Say in Interviews

**"Walk me through what you built."**
"The frontend is a working demo of the Discovery Bridge concept. Two scenarios: one proactive where the system detects a calendar change and recommends a trip modification, one reactive where the traveler requests changes via chat. Both show the BRAIN analyzing context and the BRIDGE executing across suppliers. The backend has a policy engine with a test suite that validates corporate travel rules as deterministic logic, not LLM prompts. The whole thing deploys automatically from GitHub to Azure."

**"Why didn't you build the full backend?"**
"I scoped this as a PM, not an engineer. The policy engine validates the domain model. The architecture docs specify what engineering would build. I could hand this repo to a team and they'd have typed interfaces, test cases, seed data, and a working demo to build against. That's the deliverable."

**"What would you build next?"**
"MCP server. It's the 'Plaid for AI agents' piece. Same policy engine, same domain types, but exposed as tools that any AI agent can call. I have the design ready and the tool definitions spec'd out."

**"How is this different from a Claude wrapper?"**
"The policy engine doesn't call Claude. It evaluates structured rules against typed data and returns a deterministic result. In the full system, Claude orchestrates the workflow, but when it needs a policy check, it calls this engine as a tool. Swap out Claude for any other LLM, the policy engine still works. That's infrastructure."
